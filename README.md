# AutoAgent

**A local, auditable multi-agent system for automotive failure investigation.**

AutoAgent orchestrates a pipeline of eight specialized AI agents, all running securely and locally via [Ollama](https://ollama.com/), to investigate failure scenarios, synthesize expert engineering opinions, and produce actionable engineering reports. No cloud API keys are required, ensuring complete data privacy and intellectual property protection.

## Overview

AutoAgent is designed for automotive engineers, quality assurance teams, and compliance officers who require a repeatable, traceable investigation workflow. Given a failure scenario (for example, `SC-042`), the system performs the following steps:

1. **Retrieves Context:** Searches a local knowledge base (RAG) for similar past incidents and applicable regulatory requirements.
2. **Parallel Analysis:** Runs three specialized analyses concurrently: Technical, Quality, and Compliance.
3. **Critique and Synthesize:** Cross-examines the findings to surface blind spots and unifies them into a comprehensive report.
4. **Validation:** Automatically checks report quality, retrying the synthesis step if necessary.
5. **Human-in-the-Loop:** Presents the finished report for human approval before any automated action is taken.
6. **Automation:** Upon approval, creates an engineering ticket and archives the full audit trail.

## Key Features

| Feature | Description |
|---|---|
| **100% Local Execution** | Uses Ollama with Mistral/Llama models. No data leaves the host machine. |
| **Model Context Protocol (MCP)** | Modular capability servers for Knowledge, Analysis, and Automation. |
| **LangGraph Orchestration** | Robust, graph-based workflow with type-safe state management. |
| **Full Auditability** | Every prompt, tool call, and agent decision is logged into a structured audit trail. |
| **Prompt Separation** | Agent instructions are stored as Markdown files in the `skills/` directory, allowing engineers to tune personas without modifying Python code. |

## Architecture and Workflow

The workflow is managed by **LangGraph**, which passes a strongly-typed state object (`InvestigationState`) between nodes.

```text
User Request
  │
  ▼
Orchestrator (LangGraph)
  │
  ├─► Evidence Agent      (RAG search over incidents, requirements, test reports)
  │     │
  │     ▼
  ├─► [Parallel Execution]
  │     ├─► Technical Agent   (Failure mode and root cause analysis)
  │     ├─► Quality Agent     (Process and quality system perspective)
  │     └─► Compliance Agent  (Safety standards and regulatory perspective)
  │     │
  │     ▼
  ├─► Critic Agent        (Compares and challenges the three opinions)
  │     │
  │     ▼
  ├─► Synthesis Agent     (Produces a unified final investigation report)
  │     │
  │     ▼
  ├─► Validator Agent     (Quality-checks the synthesis; retries up to 2x if needed)
  │     │
  │     ▼
  ├─► Human Approval      (Interactive approve/reject step in CLI)
  │     │
  │     ▼
  └─► Automation Agent    (Saves report to disk, creates engineering ticket)
```

Each stage writes its inputs, outputs, and decisions to the audit log, so any report generated by AutoAgent can be traced back to the exact evidence and reasoning that produced it.

## Agent Roster

| # | Agent | Role |
|---|---|---|
| 1 | **Evidence Agent** | Performs retrieval-augmented search over the local knowledge base of incidents, requirements, and test reports. |
| 2 | **Technical Agent** | Evaluates the failure from a root-cause and failure-mode engineering perspective. |
| 3 | **Quality Agent** | Assesses the finding against process and quality management system criteria. |
| 4 | **Compliance Agent** | Reviews the scenario against applicable safety standards and regulatory requirements. |
| 5 | **Critic Agent** | Cross-examines the Technical, Quality, and Compliance findings to identify gaps, contradictions, or blind spots. |
| 6 | **Synthesis Agent** | Merges all findings and critiques into a single, coherent investigation report. |
| 7 | **Validator Agent** | Checks the synthesized report against quality criteria and triggers up to two retries if it falls short. |
| 8 | **Automation Agent** | Persists the approved report and audit trail, and files the corresponding engineering ticket. |

## Prerequisites

Before installing AutoAgent, ensure the following are available on your system:

- [Ollama](https://ollama.com/) installed and running locally
- One or more compatible local models pulled (for example, `mistral` or `llama3`)
- Python 3.10 or later
- `pip` for dependency installation
- Sufficient local compute resources (CPU/GPU and RAM) to run the selected models

## Installation

1. Clone the repository:

   ```bash
   git clone https://github.com/<your-org>/autoagent.git
   cd autoagent
   ```

2. Create and activate a virtual environment:

   ```bash
   python -m venv .venv
   source .venv/bin/activate   # On Windows: .venv\Scripts\activate
   ```

3. Install dependencies:

   ```bash
   pip install -r requirements.txt
   ```

4. Pull the required Ollama models:

   ```bash
   ollama pull mistral
   ollama pull llama3
   ```

## Configuration

AutoAgent is configured through environment variables and/or a configuration file (for example, `config.yaml` or `.env`), which typically define:

- The Ollama model(s) assigned to each agent role
- The path to the local knowledge base used by the Evidence Agent
- Validation thresholds and maximum retry counts for the Validator Agent
- Output and audit-log directories
- Ticketing system integration settings for the Automation Agent

Agent personas and instructions are stored as Markdown files under the `skills/` directory, allowing prompt tuning without changes to the Python codebase.

## Usage

Run an investigation against a specific failure scenario identifier:

```bash
python main.py investigate --scenario SC-042
```

During execution, AutoAgent will:

1. Retrieve relevant evidence from the local knowledge base.
2. Run the Technical, Quality, and Compliance agents in parallel.
3. Critique and synthesize the findings into a unified report.
4. Validate the report, retrying as needed.
5. Prompt for human approval in the CLI.
6. On approval, save the report, write the audit trail, and create an engineering ticket.
